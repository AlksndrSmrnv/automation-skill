# automation-skill

Набор скиллов для **Qwen Code CLI**, помогающий писать автотесты для тест-фреймворка на
**Kotlin 2.2.20 / Java 21 / JUnit5 / Allure** (REST API, Kafka, PostgreSQL, Ignite, OpenSearch,
Kubernetes и др.).

## Что сейчас в репозитории

### `autotest-skill-generator/` — скилл-генератор
Мета-скилл: анализирует существующий проект автотестов, строит каталог атомарных возможностей и
генерирует по подробному скиллу на каждую — «как вызвать эту функцию в тесте», привязывая каждый
скилл к реальному коду (`file:line` + сниппет из живого теста). Discovery-first: не хардкодит
пути, работает с любым репозиторием.

```
autotest-skill-generator/
├── SKILL.md                          # рабочий процесс из 6 этапов
├── references/                       # discovery-guide, capability-taxonomy, quality-checklist
└── templates/                        # capability-skill, house-style-skill, capability-catalog
```

### `write-autotest/` — скилл-оркестратор
Читает тест в виде **текстовых шагов**, сопоставляет каждый шаг со сгенерированными
скиллами-возможностями и обзорным `autotest-house-style`, находит пробелы в данных, задаёт точечные
вопросы и пишет готовую реализацию на Kotlin/JUnit5/Allure **точно в стиле проекта**. Ничего не
выдумывает: API берётся из скиллов и кода, реальные значения не хардкодятся (плейсхолдеры/конфиг).

```
write-autotest/
├── SKILL.md                          # рабочий процесс из 8 этапов
└── references/                       # step-to-skill-mapping, gap-analysis, test-assembly
```

### `.qwen/agents/framework-explorer.md` — read-only сабагент
Тонкий анализатор тест-фреймворка: по заданной зоне (слои/конфиг/тест-классы) возвращает пути +
выдержки + наблюдения, ничего не выдумывая. Используется и генератором (картирование), и
оркестратором (анализ проекта), когда сабагенты Qwen доступны; иначе скиллы анализируют сами.

## Установка в Qwen Code

Скиллы Qwen живут в одном из двух мест:
- `~/.qwen/skills/<имя>/SKILL.md` — персональные, доступны во всех проектах;
- `.qwen/skills/<имя>/SKILL.md` — внутри репозитория, шарятся через git.

Сабагенты — аналогично: `~/.qwen/agents/<имя>.md` (персонально) или `.qwen/agents/<имя>.md`
(в репозитории).

Установить нужно **оба скилла и сабагент**. Удобно класть их персонально (доступны во всех
проектах автотестов):

```bash
mkdir -p ~/.qwen/skills ~/.qwen/agents
cp -r autotest-skill-generator write-autotest ~/.qwen/skills/
cp .qwen/agents/framework-explorer.md ~/.qwen/agents/
```

Либо положить их прямо в репозиторий автотестов, чтобы шарить через git (туда же генератор пишет
`autotest-house-style` и скиллы-возможности):

```bash
# из корня репозитория автотестов
mkdir -p .qwen/skills .qwen/agents
cp -r /path/to/automation-skill/autotest-skill-generator /path/to/automation-skill/write-autotest .qwen/skills/
cp /path/to/automation-skill/.qwen/agents/framework-explorer.md .qwen/agents/
```

Проверь, что Qwen их видит: команда `/skills` должна показать `autotest-skill-generator` и
`write-autotest`, а `/agents` — `framework-explorer` (сабагент опционален: без него скиллы
анализируют проект сами через read/grep/glob).

> Порядок использования: сначала `autotest-skill-generator` строит библиотеку
> (`autotest-house-style` + скиллы-возможности), затем `write-autotest` пишет по ней тесты. Чтобы
> `write-autotest` работал, он и сгенерированные скиллы должны быть видны в одном `/skills`.

## Использование

В репозитории автотестов запусти Qwen Code и вызови скилл явно через `/skills
autotest-skill-generator` или просто попроси:

```
Сгенерируй скиллы по нашему тест-фреймворку. Корень проекта автотестов: <путь>.
```

Генератор:
1. построит карту проекта (стек, слои, базовый класс, конфиг, DI, Allure, ассерты);
2. соберёт каталог атомарных возможностей в `.qwen/skill-catalog/_catalog.md` (вне skills-root);
3. задаст уточняющие вопросы (что генерировать, что упущено, куда писать);
4. создаст по скиллу на возможность + обзорный `autotest-house-style` в `.qwen/skills/`;
5. проверит результат по чек-листу;
6. выдаст итоговый отчёт (путь каталога, созданные/исключённые скиллы, TODO, что проверено).

После генерации Qwen увидит новые скиллы в `/skills` и будет подключать их по `description`,
когда вы пишете соответствующий шаг теста.

## Под Qwen Code (важно)

Скиллы рассчитаны на механику Qwen Code, а не Claude Code:
- frontmatter: обязательны только `name` и `description`; опционально `paths:` (glob-гейтинг);
- срабатывание model-invoked по `description` + явный вызов `/skills <имя>`;
- сабагенты: нет Claude-специфичных `Explore`/`general-purpose`, но Qwen-сабагенты
  поддерживаются (через `/agents` и Task Tool, `.qwen/agents/`) — используются, если доступны,
  иначе анализ ведётся инструментами read/grep/glob/shell;
- вопросы пользователю: вместо Claude-style `AskUserQuestion` используется `ask_user_question`,
  если доступен; иначе вопросы задаются обычным текстом.

## Использование `write-autotest`

В репозитории автотестов (там же, где лежат `autotest-house-style` и скиллы-возможности) вызови
`/skills write-autotest` или просто опиши задачу шагами:

```
Напиши автотест по шагам:
1. <шаг>
2. <шаг>
Ожидаемый результат: <что должно произойти>.
```

Оркестратор:
1. зафиксирует цель и шаги;
2. загрузит `autotest-house-style` и проверит наличие скиллов-возможностей (если их нет — предложит
   сначала запустить `autotest-skill-generator`);
3. сопоставит шаги со скиллами и отметит непокрытые;
4. проанализирует проект (куда положить тест, базовый класс, образцовый аналог);
5. проведёт gap-анализ достаточности данных;
6. при критических пробелах задаст точечные вопросы и не будет писать тест, пока они не закрыты;
7. напишет тест в стиле проекта (реальное API из скиллов, плейсхолдеры вместо секретов);
8. сделает самопроверку/компиляцию и выдаст итоговый отчёт.

## Роадмап (следующие итерации)

- Расширение сабагентов в `.qwen/agents/` (специализация по подсистемам поверх `framework-explorer`).
- Накопление библиотеки скиллов-возможностей под конкретные проекты автотестов.

Генератор и оркестратор уже стоят на общей почве: единый шаблон скиллов, обзорный
`autotest-house-style` (грузится первым), перекрёстные ссылки по имени и общий сабагент
`framework-explorer`.
