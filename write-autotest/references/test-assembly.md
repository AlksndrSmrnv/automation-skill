# Test Assembly — как собрать тест из возможностей в стиле проекта (Этап 7–8)

Цель этапа — превратить таблицу сопоставления (Этап 3) и закрытые данные (Этап 5–6) в готовый
тест-класс, **неотличимый по стилю** от существующих тестов проекта. Источники кода — только
`autotest-house-style` (каркас) и скиллы-возможности (сниппеты действий). Ничего не выдумывай;
реальные значения не хардкодь.

## 1. Старт от скелета house-style

Открой `autotest-house-style` и возьми из него каркас (он уже снят с реального кода проекта):
- **базовый класс** и расширения — от чего наследуется тест (`@ExtendWith`, `*IntegrationTest`);
- **DI** — как тест получает степы/клиенты (Spring `@Autowired` / ручное создание / object);
- **Allure-разметку класса** — `@Epic/@Feature/@Story/@DisplayName` как принято;
- **жизненный цикл** — что в `@BeforeEach/@AfterEach` (setup/cleanup);
- **идиому ассертов** — AssertJ / Kotest / JUnit;
- **async** — Awaitility/поллинг, как принято ждать результата;
- **тестовые данные** — билдеры/фабрики/генераторы.

Скелет типичного теста из house-style — твой шаблон структуры. Не отклоняйся от него.

## 2. Размести файл теста

По итогам анализа проекта (Этап 4):
- **пакет/модуль** — туда, где лежат тесты этой подсистемы/домена (рядом с образцовым аналогом);
- **имя класса** — по конвенции проекта (`*Test`/`*IT`/`*Spec`);
- **имя метода и `@DisplayName`** — отражают цель теста из Этапа 1, в стиле существующих тестов;
- **теги/категории** (`@Tag`) — как у аналогичных тестов.

## 3. Собери тело теста из шагов

Иди по таблице сопоставления в порядке arrange → act → assert. Для каждого шага:
1. Открой его скилл-возможность, возьми сниппет из раздела «Минимальный пример в тесте».
2. Адаптируй: подставь реальные аргументы (значения из Этапа 5–6 или билдеры), убери лишний контекст.
3. Получи зависимость так, как показывает скилл и house-style (DI/создание) — единообразно в классе.
4. Оберни в Allure-шаг так, как принято (`@Step`-методы степов и/или `Allure.step("…") { … }`).

Структура тела:
- **arrange** — предусловия и тест-данные (seed, подготовка топика/сущности, билдеры);
- **act** — основное действие сценария;
- **assert** — проверки идиомой проекта; для async-результатов используй ожидание из house-style
  (`await().untilAsserted { … }` и т.п.), а не «сырой» вызов сразу после действия.

Связки «действие → проверка» бери из связанных скиллов (например, `kafka-send-record` →
`kafka-consume-await`): отправил — дождись/проверь, а не предполагай мгновенный результат.

## 4. Очистка и идемпотентность

- Добавь cleanup так, как делает проект (`@AfterEach`/общий extension): убери созданные строки,
  откати изменения конфига пода, сбрось состояние, чтобы не ломать соседние тесты.
- Проверь идемпотентность: тест должен проходить при повторном запуске (не зависеть от данных,
  оставшихся от прошлого прогона).

## 5. Безопасность в коде теста

- Никаких секретов/реальных значений: токены, пароли, ключи, хосты/URL стендов, IP, namespace,
  имена подов, e-mail, PII — **не хардкодить**.
- Используй плейсхолдеры или ключи конфига/переменные окружения (`${DB_PASSWORD}`,
  `kafka.bootstrap-servers`, `<BASE_URL>`), как это делает house-style и скиллы.
- Если сомневаешься, секрет ли значение, — выноси его в конфиг/плейсхолдер.

## 6. Непокрытые шаги

Шаги со статусом «частично»/«не покрыт» (Этап 3), оставшиеся без реализации, оформи **явным TODO**,
а не фиктивным вызовом:

```kotlin
// TODO: шаг 4 «проверить алерт в системе X» — нет скилла-возможности.
//       Сгенерировать скилл через autotest-skill-generator или уточнить способ проверки.
```

Каждый такой TODO попадает в итоговый отчёт (раздел «непокрытые шаги»/«открытые TODO»).

## 7. Чек-лист самопроверки (Этап 8)

Пройди по готовому тесту (стиль `quality-checklist.md` генератора):

**Достоверность:**
- [ ] Все вызванные методы/классы реально существуют (сверено со скиллами/кодом), сигнатуры верны.
- [ ] Импорты настоящие (корневой пакет проекта), без выдуманных классов.
- [ ] Способ получения зависимостей соответствует house-style (DI/создание).

**Стиль:**
- [ ] Базовый класс, аннотации класса, теги — как в проекте.
- [ ] Allure-разметка на месте (шаги, `@DisplayName`), как принято.
- [ ] Ассерты — идиомой проекта; async-результаты проверяются через ожидание, а не сразу.
- [ ] Есть очистка/cleanup, тест идемпотентен.

**Безопасность и полнота:**
- [ ] В коде нет секретов/реальных значений — только плейсхолдеры/ключи конфига.
- [ ] Все покрытые шаги реализованы; непокрытые помечены `// TODO:`.
- [ ] Тест-данные и ожидаемые результаты соответствуют ответам из gap-анализа (Этап 5–6).

## 8. Компиляция/прогон (если окружение позволяет)

- Gradle: `./gradlew test --tests "<FQN.testMethod>"` (или модульная задача).
- Maven: `mvn -Dtest=<Class#method> test`.
- Если зависимостей/стенда нет или прогон невозможен — **не выдавай зелёный результат**: отметь, что
  проверка только статическая (самопроверка по чек-листу), и укажи это в отчёте.

## 9. Итоговый отчёт (формат)

Краткий отчёт пользователю в конце:

```
Тест создан: <путь к файлу> — класс <Имя>, метод <имя> (@DisplayName "<...>")
Использованные скиллы: autotest-house-style, <skill-1>, <skill-2>, …
Непокрытые шаги: шаг N «<текст>» — <причина> (TODO в коде)
Открытые TODO: <список или «нет»>
Проверено: <компиляция/прогон: команда + результат | только самопроверка по чек-листу>
```
