# Step-to-Skill Mapping — как сопоставить текстовые шаги со скиллами (Этап 3)

Цель этапа — для каждого текстового шага найти скилл(ы)-возможность, по которым его можно
реализовать, и честно отметить шаги, для которых скилла нет. Результат — таблица сопоставления, на
которую опираются gap-анализ (Этап 5) и сборка теста (Этап 7).

Главное правило: **если для шага нет скилла — не выдумывай вызов**. Помечай шаг как непокрытый;
дальше он станет вопросом пользователю, TODO в коде или запросом нового скилла у генератора.

## 1. Узнай, какие скиллы доступны

- Посмотри список скиллов: `/skills` в Qwen Code. Тебя интересуют `autotest-house-style` и
  скиллы-возможности (обычно по схеме `<подсистема>-<действие>`: `kafka-send-record`,
  `rest-get-with-auth`, `postgres-run-query`, `opensearch-await-log`, `k8s-restart-deployment` и т.п.).
- Сопоставление идёт по `description`: Qwen model-invoked подключает скилл по тому, как
  сформулирован шаг. Читай `description` скилла как ответ на вопрос «какой шаг теста это покрывает».
- Если в репозитории есть каталог возможностей (`.qwen/skill-catalog/_catalog.md` от генератора) —
  он даёт быстрый обзор: подсистема → возможность → имя скилла. Используй его как карту.

## 2. Декомпозируй составные шаги

Один текстовый шаг часто = несколько атомов. Разбей его до уровня скиллов-возможностей:

- «Отправь сообщение в Kafka и убедись, что оно обработано» →
  `kafka-send-record` (act) + `kafka-consume-await` или `postgres-assert-row` (assert).
- «Создай пользователя через API и проверь его в БД» →
  `rest-post-json` (act) + `postgres-run-query`/`postgres-assert-row` (assert).
- «Перезапусти под и дождись готовности» →
  `k8s-restart-deployment` (act) + `k8s-wait-ready` (assert/await).

И наоборот: несколько мелких текстовых шагов могут покрываться одним скиллом (если в проекте это
один step-метод). Ориентируйся на атомы из библиотеки скиллов, а не на дословную разбивку текста.

## 3. Классифицируй шаг по роли

Отметь для каждого шага его роль в тесте — это пригодится при сборке (arrange/act/assert):

- **arrange / предусловие** — подготовка данных и окружения (seed в БД, создать топик, выставить
  конфиг, залогиниться). Часто это setup-атомы без осмысленного возврата.
- **act / действие** — основное действие сценария (отправить, вызвать API, рестартнуть).
- **assert / проверка** — проверка результата (статус-код, строка в БД, лог в OpenSearch, значение
  в кэше). Здесь важна идиома ассертов и async-ожидание из `autotest-house-style`.

## 4. Привяжи шаг к подсистеме и скиллу

Определи подсистему по содержанию шага и техническим словам, затем подбери скилл:

| Звучание шага | Подсистема | Кандидаты-скиллы (имена зависят от проекта) |
|---------------|-----------|---------------------------------------------|
| «вызови GET/POST эндпоинт», «дёрни API», «получи токен» | REST | `rest-get-with-auth`, `rest-post-json`, `auth-get-token` |
| «отправь в топик», «дождись сообщения из Kafka» | Kafka | `kafka-send-record`, `kafka-consume-await` |
| «выполни SQL», «проверь строку в БД», «засидь данные» | PostgreSQL | `postgres-run-query`, `postgres-assert-row` |
| «положи/возьми из кэша», «дождись значения в Ignite» | Ignite | `ignite-cache-put`, `ignite-cache-get` |
| «найди лог», «дождись записи в логах» | OpenSearch | `opensearch-find-log`, `opensearch-await-log` |
| «логи пода», «рестарт деплоймента», «patch конфига» | Kubernetes | `k8s-get-pod-logs`, `k8s-restart-deployment`, `k8s-patch-config` |

Это ориентир, а не закрытый список — подсистемы за пределами таблицы (gRPC, Redis, S3/MinIO и т.п.)
ищи теми же признаками: техническое слово в шаге ↔ скилл с подходящим `description`.

## 5. Зафиксируй таблицу сопоставления

Держи результат в рабочем контексте (в чат целиком не вываливай — давай сводку). Формат:

| № шага | Текст шага | Роль | Подсистема | Скилл(ы) | Статус |
|--------|-----------|------|-----------|----------|--------|
| 1 | Засидь пользователя в БД | arrange | PostgreSQL | `postgres-run-query` | покрыт |
| 2 | Отправь событие в топик | act | Kafka | `kafka-send-record` | покрыт |
| 3 | Дождись обработки и проверь статус в БД | assert | Kafka/PostgreSQL | `kafka-consume-await` + `postgres-assert-row` | покрыт |
| 4 | Проверь алерт в системе X | assert | — | — | не покрыт |

Статусы:
- **покрыт** — есть скилл(ы), реализующий шаг целиком.
- **частично** — скилл закрывает часть шага (например, действие есть, а специфичная проверка — нет).
- **не покрыт** — подходящего скилла нет.

## 6. Что делать с непокрытыми шагами

Для шагов со статусом «частично» / «не покрыт»:
- если не хватает данных или непонятна суть — это вход в gap-анализ (Этап 5) и вопрос пользователю;
- если действие в проекте есть, но скилл на него не сгенерирован — предложи пользователю запустить
  `autotest-skill-generator`, чтобы добавить скилл (так тест останется в стиле проекта);
- если шаг останется без реализации — в коде теста это явный `// TODO:` с пояснением (не фиктивный
  вызов) и строка в итоговом отчёте (Этап 8).

С готовой таблицей переходи к анализу проекта (Этап 4) и gap-анализу (`gap-analysis.md`, Этап 5).
