# Gap Analysis — достаточно ли данных, чтобы писать тест (Этап 5–6)

Цель этапа — до написания убедиться, что данных хватает на корректный тест, и закрыть критические
пробелы вопросами к пользователю. Это самая дешёвая точка коррекции: вопрос стоит дешевле, чем
тест с выдуманными значениями или неверной проверкой.

Правило: **не выдумывай недостающее**. Если данных нет — либо найди их в коде/house-style, либо
спроси (критический пробел), либо оставь TODO (некритический пробел).

## Чек-лист достаточности

Пройди по каждой категории применительно к шагам из таблицы сопоставления (Этап 3). Для каждого
пункта отметь: данные есть / нет / видно из кода.

**1. Тест-данные (вход).**
- [ ] Известны конкретные значения входа для act/arrange-шагов (тело запроса, ключ/значение
      сообщения, параметры SQL, id сущностей)? Или есть билдер/фабрика из house-style, который их
      порождает?
- [ ] Понятно, что в значениях — фиксированное, а что генерируется (рандом/последовательность)?

**2. Ожидаемые результаты (ассерты).**
- [ ] Для каждого assert-шага ясно, **что** проверяем и **с чем** сравниваем (ожидаемый статус-код,
      значение поля, число строк, наличие/отсутствие лога)?
- [ ] Известна строгость проверки (точное равенство, содержит, по подмножеству полей)?

**3. Окружение и конфиг.**
- [ ] Известно, какие ключи конфига/переменные окружения нужны шагам (из раздела «конфиг/env»
      соответствующих скиллов-возможностей)?
- [ ] Понятно, что поднимается локально (Testcontainers), а что — внешний стенд? (берётся из
      `autotest-house-style`.) Реальные значения не нужны — нужны имена ключей/контейнеров.

**4. Предусловия.**
- [ ] Ясно состояние системы перед тестом (нужен ли seed данных, существующая сущность,
      залогиненный пользователь, поднятый топик)?
- [ ] Эти предусловия закрываются arrange-шагами/скиллами или их надо добавить?

**5. Очистка после теста.**
- [ ] Понятно, что нужно почистить после теста (созданные строки, отправленные сообщения,
      изменённый конфиг пода) и как (`@AfterEach`/cleanup по house-style)?
- [ ] Нет ли побочных эффектов, которые сломают соседние тесты, если их не убрать?

**6. Непокрытые шаги.**
- [ ] Для шагов со статусом «частично»/«не покрыт» (Этап 3) понятно, что делать: спросить, оставить
      TODO или попросить сгенерировать скилл?

## Классификация пробелов

Каждый незакрытый пункт отнеси к одному из двух типов:

- **Критический** — без него тест нельзя написать корректно: непонятен ожидаемый результат, нет
  значений для основного действия, неясно предусловие, от которого зависит проверка. Критические
  пробелы **блокируют Этап 7** — сначала спроси (Этап 6).
- **Некритический** — можно оформить TODO или разумным плейсхолдером, не искажая суть теста:
  второстепенное поле, необязательная очистка, шаг вне основного сценария. Допустимо оставить в
  коде явным `// TODO:` и упомянуть в отчёте (Этап 8).

Если сомневаешься, критический пробел или нет, — считай критическим и спроси.

## Сначала посмотри, потом спрашивай

Не задавай вопросов, ответ на которые виден из репозитория или house-style. Перед вопросом проверь:
- образцовый тест-аналог (Этап 4) — какие значения и проверки он использует;
- `autotest-house-style` — конфиг, окружение, идиома ассертов, билдеры данных, очистка;
- скиллы-возможности — раздел «параметры / нужный конфиг и env» и «возвращает и как проверять».

Спрашивай только то, что действительно знает лишь пользователь (бизнес-значения, ожидаемые
результаты конкретного кейса, какой стенд/профиль использовать).

## Как задавать вопросы (Этап 6)

- Инструмент: `ask_user_question`, если доступен в Qwen; иначе обычный текст. **Claude-style
  `AskUserQuestion` не используй.**
- Группируй родственные вопросы, формулируй конкретно, по возможности с вариантами ответа.
- Привязывай вопрос к шагу: «Шаг 3 (проверка в БД): с каким значением статуса сравнивать —
  `PROCESSED` или другим?».
- Не задавай за раз больше, чем нужно для разблокировки: сфокусируйся на критических пробелах.

Примеры точечных вопросов:
- «Какой ожидаемый статус-код у запроса из шага 2 — 200 или 201?»
- «Шаг 1 (seed): использовать существующего тестового пользователя из билдера или создавать нового?»
- «Для проверки лога (шаг 4): по какому полю/корреляционному id искать запись?»
- «Нужно ли чистить созданную в шаге 1 строку после теста, или это делает общий cleanup?»

## После ответов — повтори gap-анализ

Получив ответы, **вернись к чек-листу** и переотметь пункты. Повторяй цикл «gap-анализ → вопросы»,
пока не закрыты все **критические** пробелы. Только после этого переходи к написанию теста
(`test-assembly.md`, Этап 7). Оставшиеся некритические пробелы зафиксируй как TODO для кода и отчёта.
