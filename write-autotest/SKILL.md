# Write Autotest — оркестратор написания теста по шагам

Этот скилл берёт тест в виде **текстовых шагов** и пишет его реализацию **точно в стиле проекта**,
опираясь на библиотеку скиллов, которую создал `autotest-skill-generator`: обзорный
`autotest-house-style` (каркас теста) и скиллы-возможности (как вызвать конкретное действие).
Сам оркестратор не знает API проекта — он берёт его из этих скиллов и из кода репозитория.

**Главный принцип: ничего не выдумывай.** Каждый вызов в тесте берётся из скилла-возможности или
из реального кода проекта; сигнатуры, импорты и Allure-разметка — как в проекте; реальные значения
(хосты, токены, id стендов) не хардкодятся — только плейсхолдеры или ключи конфига. Если для шага
нет скилла или не хватает данных — лучше задать вопрос или пометить TODO, чем сочинить вызов.

## Платформа: Qwen Code CLI

Скилл рассчитан на **Qwen Code CLI** (не Claude Code). Учитывай особенности Qwen:

- **Где живут скиллы:** `~/.qwen/skills/<имя>/SKILL.md` (персональные) или
  `.qwen/skills/<имя>/SKILL.md` (внутри репозитория, шарятся через git). Скиллы-возможности и
  `autotest-house-style` обычно лежат в `.qwen/skills/` репозитория тестов — оттуда Qwen видит их сам.
- **Frontmatter:** обязательны только `name` (маска `^[\p{L}\p{N}_:.-]+$`, без пробелов) и
  `description`. Опционально `paths:` — glob-гейтинг.
- **Срабатывание:** скиллы model-invoked — Qwen сам подключает скилл по `description`; плюс явный
  вызов `/skills <имя>`. Чтобы найти нужный скилл-возможность под шаг, читай его `description`.
- **Сабагенты:** в Qwen нет Claude-специфичных `Explore`/`general-purpose`, но свои сабагенты
  поддерживаются (через `/agents` и Task Tool, конфиги в `.qwen/agents/`). Если доступен
  `framework-explorer` (или похожий) — делегируй ему разбор проекта; иначе анализируй сам
  инструментами read/grep/glob/shell.
- **Вопросы пользователю:** не используй Claude-style `AskUserQuestion`. Применяй
  `ask_user_question`, если он доступен; иначе спрашивай обычным текстом.
- **Содержимое на русском; код, имена символов и импорты — английские, как в проекте.**

## Безопасность (обязательно)

Никогда не переноси в код теста секреты и чувствительные значения: токены, пароли, ключи, реальные
хосты/URL стендов, IP, namespace/имена подов, e-mail, персональные/внутренние данные. Везде
используй плейсхолдеры (`<TOKEN>`, `<BASE_URL>`) или ключи конфига/переменные окружения
(`kafka.bootstrap-servers`, `${DB_PASSWORD}`). Если сомневаешься, секрет ли это, — считай, что да.

## Предусловия

1. **Библиотека скиллов существует.** Нужны обзорный `autotest-house-style` и скиллы-возможности
   под действия проекта. Их создаёт `autotest-skill-generator`. Если их нет (см. Этап 2) — не пиши
   тест вслепую: предупреди и предложи сначала запустить генератор.
2. **Доступ к репозиторию тестов.** Нужен корень проекта автотестов, чтобы найти целевой пакет,
   базовый класс и образцовый тест-аналог. Спроси путь, если он не очевиден из рабочей директории.
3. **Понятные шаги.** Тест задан текстовыми шагами. Если шаги размыты — уточни цель и шаги до начала.

## Рабочий процесс (8 этапов)

Не перескакивай к написанию теста, пока не сопоставил шаги со скиллами и не закрыл критические
пробелы в данных. Поспешность здесь = тест с выдуманным API или хардкодом значений.

### Этап 1. Вход: цель и шаги

Зафиксируй, **что** проверяет тест (цель/ожидаемый итог) и **пронумерованные шаги**. Если
пользователь дал сплошной текст — раздели его на дискретные шаги и подтверди нумерацию. Эта
нумерация — основа таблицы сопоставления (Этап 3) и gap-анализа (Этап 5). Не начинай Этап 2, пока
шаги не ясны.

### Этап 2. Загрузка house-style и проверка предусловий

Первым подключи `autotest-house-style` (через `/skills autotest-house-style` или дай Qwen
подтянуть его по описанию). Он задаёт каркас: базовый тест-класс, DI, конфиг/окружение,
Allure-разметку, идиому ассертов, тестовые данные, жизненный цикл и скелет типичного теста.

Проверь, что в `/skills` есть и скиллы-возможности (по подсистемам, которые встречаются в шагах).
Если `autotest-house-style` или скиллов-возможностей нет — **остановись и сообщи пользователю**:
без них тест получится «по памяти» и не совпадёт со стилем проекта. Предложи сначала запустить
`autotest-skill-generator`, чтобы построить библиотеку скиллов.

### Этап 3. Сопоставление шагов со скиллами

Для каждого шага из Этапа 1 подбери скилл(ы)-возможность по `description` (как звучит шаг ↔ что
делает скилл). Действуй по **`references/step-to-skill-mapping.md`**: декомпозируй составные шаги,
учитывай, что один шаг может требовать нескольких скиллов (отправил → дождись), различай
arrange/act/assert. Итог — таблица `№ шага | текст | подсистема | скилл(ы) | статус` со статусами
покрыт / частично / не покрыт. Шаги без скилла явно пометь — это кандидаты в вопросы, TODO или в
запрос нового скилла у генератора. **Не выдумывай вызов для непокрытого шага.**

### Этап 4. Анализ проекта

Нужно понять, **куда и как** встроить тест: целевой пакет/модуль, базовый класс, образцовый
тест-аналог (похожий по подсистемам), конвенции именования класса/метода/`@DisplayName`. Опирайся
на `autotest-house-style`; недостающее ищи в репозитории.

Это широкий поиск по файлам. Если в Qwen доступны сабагенты (`/agents`, Task Tool, `.qwen/agents/`)
и есть `framework-explorer` — делегируй ему разбор зоны и попроси вернуть пути + выдержки +
наблюдения. Если сабагентов нет — разбирай сам инструментами read/grep/glob/shell. Зафиксируй:
куда положить файл теста, от чего наследоваться, какой тест взять за образец.

### Этап 5. Gap-анализ (ключевой этап)

Прежде чем писать, проверь, **хватает ли данных**. Пройди чек-лист из
**`references/gap-analysis.md`** по категориям: конкретные тест-данные (значения входа), ожидаемые
результаты/ассерты (что и с чем сравнивать), окружение/конфиг (ключи/контейнеры/стенды),
предусловия (состояние перед тестом), очистка (что чистить после), шаги без скилла. Каждый пробел
классифицируй как **критический** (без него тест не написать корректно) или **некритический**
(можно оформить TODO/плейсхолдером).

### Этап 6. Вопросы при пробелах

Если есть **критические** пробелы — **НЕ пиши тест**. Задай точечные вопросы (`ask_user_question`,
если доступен; иначе обычным текстом — Claude-style `AskUserQuestion` не используй). Спрашивай
только то, чего не видно из кода/house-style: сначала посмотри сам, потом спрашивай. Формулируй
вопросы сгруппированно и конкретно (см. `references/gap-analysis.md`). После ответов **повтори
Этап 5**, пока критические пробелы не закрыты. Некритические пробелы допустимо оставить как TODO.

### Этап 7. Написание теста

Собери тест по **`references/test-assembly.md`**. Старт — скелет из `autotest-house-style`
(базовый класс, DI, Allure, жизненный цикл, идиома ассертов, async). Размести файл в пакете/модуле
из Этапа 4 по конвенциям именования. Для каждого сопоставленного шага возьми сниппет из его
скилла-возможности, адаптируй аргументы и способ получения зависимостей; выстрой arrange/act/assert
с Allure-шагами; обработай async/await и очистку (`@AfterEach`/cleanup) как в house-style.

Не выдумывай API, которого нет в скиллах/коде. Реальные значения не хардкодь — плейсхолдеры/ключи
конфига. Непокрытые шаги (Этап 3) оформи явными `// TODO:` с пояснением, а не фиктивным вызовом.

### Этап 8. Самопроверка и отчёт

Пройди чек-лист самопроверки из `references/test-assembly.md`: импорты реальны, вызванное API
существует, Allure-разметка на месте, ассерты идиоматичны, секретов нет, непокрытые шаги помечены
TODO. Если окружение позволяет — попробуй компиляцию/прогон (`./gradlew test --tests …` или
`mvn -Dtest=…`); если нет — честно отметь, что проверка только статическая.

Затем выдай краткий **итоговый отчёт**:
- **созданный тест** — путь к файлу, имя класса/метода;
- **использованные скиллы** — список имён (включая `autotest-house-style`);
- **непокрытые шаги** — какие шаги остались без скилла/реализации и почему;
- **открытые TODO** — где не хватило данных или нужна проверка пользователем;
- **что проверено** — компиляция/прогон или только самопроверка; результат.

## Чего не делать

- Не писать тест без `autotest-house-style` и скиллов-возможностей — сначала запусти генератор.
- Не выдумывать сигнатуры, импорты, API, конфиг — только то, что есть в скиллах и репозитории.
- Не хардкодить секреты и реальные значения стендов — плейсхолдеры/ключи конфига.
- Не писать тест, пока есть незакрытые **критические** пробелы (Этап 5–6).
- Не подменять непокрытый шаг фиктивным вызовом — помечай TODO и сообщай в отчёте.
- Не использовать слой, который проект не вызывает (сырые клиенты вместо степов) — следуй house-style.
- Не вываливать промежуточные таблицы/анализ целиком в чат без нужды — держи в работе, в чат давай
  сводку и вопросы.

## Структура этого скилла

```
write-autotest/
├── SKILL.md                       ← ты здесь (рабочий процесс из 8 этапов)
└── references/
    ├── step-to-skill-mapping.md   ← как сопоставлять текстовые шаги со скиллами (Этап 3)
    ├── gap-analysis.md            ← чек-лист достаточности данных и какие вопросы задавать (Этап 5–6)
    └── test-assembly.md           ← как собрать тест из возможностей в стиле house-style (Этап 7–8)
```

## Связанные скиллы и сабагенты

- `autotest-house-style` — каркас теста; грузи первым (Этап 2).
- `autotest-skill-generator` — создаёт библиотеку скиллов-возможностей; запусти, если их нет.
- Скиллы-возможности (`kafka-send-record`, `rest-get-with-auth`, `postgres-run-query`, … — состав
  зависит от проекта) — источник сниппетов для шагов.
- Сабагент `framework-explorer` (`.qwen/agents/`, если доступен) — разбор проекта на Этапе 4.

## Как запустить (в Qwen Code CLI)

1. Установи скилл: скопируй папку `write-autotest/` в `~/.qwen/skills/` или в `.qwen/skills/`
   репозитория тестов (туда же, где лежат `autotest-house-style` и скиллы-возможности).
2. В репозитории тестов вызови скилл явно `/skills write-autotest` или просто опиши задачу:

```
Напиши автотест по шагам:
1. <шаг>
2. <шаг>
...
Ожидаемый результат: <что должно произойти>.
```

Ожидаемый результат: зафиксированы цель и шаги → загружен `autotest-house-style` → шаги сопоставлены
со скиллами → проанализирован проект (куда класть тест) → пройден gap-анализ и (при необходимости)
заданы вопросы → написан тест в стиле проекта с реальным API и плейсхолдерами вместо секретов →
выполнена самопроверка/компиляция → выдан итоговый отчёт (Этап 8).
