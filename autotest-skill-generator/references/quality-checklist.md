# Quality Checklist — самопроверка сгенерированных скиллов (Этап 5)

Пройди по каждому сгенерированному скиллу и по набору в целом. Цель — убедиться, что по скиллу
можно написать шаг теста **не открывая исходники**, и что в нём нет выдумок. Исправь всё найденное
до завершения работы.

## По каждому скиллу-возможности

**Достоверность (самое важное):**
- [ ] Каждая ссылка `file:line` существует и указывает на тот символ, о котором речь (перепроверь
      чтением файла — строки могли сместиться).
- [ ] Точка входа (`Класс.метод`) реально существует с такой сигнатурой; имена параметров и типы
      совпадают с кодом.
- [ ] Импорты в сниппете настоящие (корневой пакет проекта из Этапа 1), без выдуманных классов.
- [ ] Сниппет снят с реального теста и работоспособен в стиле проекта (не псевдокод).
- [ ] Способ получения объекта (DI/создание) соответствует тому, как делает проект (Этап 1 §7).

**Полнота:**
- [ ] Заполнены все секции шаблона; нет оставшихся плейсхолдеров `<...>`.
- [ ] Указан нужный конфиг/env; читатель поймёт, что должно быть настроено.
- [ ] Описано, что метод возвращает и как это проверять — идиомой ассертов проекта (Этап 1 §9).
- [ ] Раздел «Подводные камни» отражает реальное поведение (таймауты, ретраи, async, очистка),
      а не общие слова.

**Триггеринг и связи:**
- [ ] `description` во frontmatter «пушистая» и конкретная: что делает + когда применять +
      триггер-фразы (RU и ключевые EN-слова). Qwen model-invoked подключает скилл по description —
      он должен срабатывать на естественную формулировку шага теста.
- [ ] `name` соответствует схеме `<подсистема>-<действие>`, маске `^[\p{L}\p{N}_:.-]+$` (без
      пробелов) и совпадает с именем папки. Только `name`+`description` обязательны; `paths:` —
      опционально.
- [ ] Проставлены перекрёстные ссылки по имени на родственные скиллы (например, `kafka-send-record`
      ↔ `kafka-consume-await`) и на `autotest-house-style` (в Qwen wiki-ссылок нет — указывай имена).
- [ ] Содержимое на русском; код/символы/импорты — английские, как в проекте.

## По набору в целом

- [ ] Создан обзорный скилл `autotest-house-style` и на него ссылаются остальные.
- [ ] Покрыты все возможности, подтверждённые пользователем на Этапе 3 (сверься с `_catalog.md`).
- [ ] Нет дублей: одна возможность = один скилл; пересекающиеся обёртки сведены к используемой.
- [ ] Имена уникальны, папки разложены как `<output>/<имя-скилла>/SKILL.md`.
- [ ] Если у скилла есть длинные примеры/детали — они вынесены в его собственный `references/`
      (тот же progressive disclosure), а `SKILL.md` остаётся обозримым.

## Мысленный прогон (рекомендуется на 1–2 скиллах)

Представь, что ты оркестратор и тебе дан шаг теста, требующий этой возможности. Хватает ли скилла,
чтобы написать корректный Kotlin-код без чтения исходников? Если нет — чего не хватило (сигнатуры,
конфига, способа проверки, очистки)? Допиши это в скилл.

## Если данных не хватило

Если по какой-то возможности ты не смог достоверно заполнить шаблон (не нашёл пример теста, не
уверен в сигнатуре) — **не выдумывай**. Либо пометь скилл как черновик с явным TODO и перечисли,
что нужно уточнить, либо вынеси возможность из набора и сообщи пользователю. Достоверность важнее
полноты.
