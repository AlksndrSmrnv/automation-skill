# Capability Taxonomy — что считать «атомарной возможностью» (Этап 2)

Атомарная возможность — это **одно осмысленное действие, которое тест выполняет над подсистемой
и переиспользует**. Из таких атомов оркестратор потом собирает шаги теста. Этот файл помогает
выделять атомы консистентно по всем подсистемам и не дробить/не укрупнять лишнего.

## Правила выделения атома

1. **Атом виден из теста.** Если действие реально вызывается в тестах (через слой, выявленный на
   Этапе 1) — это кандидат. Внутренние/приватные методы, до которых тесты не дотягиваются, не
   документируем.
2. **Атом самодостаточен.** По нему можно написать шаг теста без чтения исходников: понятны вход,
   выход и как проверять результат.
3. **Не дробить ниже полезного.** «Открыть соединение» + «выполнить запрос» + «закрыть» — это
   обычно один атом «выполнить SQL и получить строки», если тесты вызывают именно обёртку.
4. **Не укрупнять до сценария.** Целый бизнес-сценарий из 5 шагов — не атом. Атом — кирпичик
   («отправить запись в Kafka»), из которого сценарий собирается.
5. **Уровень = слой тестов.** Документируй на том слое, который вызывают тесты (см. Этап 1 §4).
6. **Без дублей.** Если несколько обёрток делают одно и то же — бери ту, что используют тесты.

## Атомы по подсистемам

Ниже — ориентиры. Конкретные имена методов берутся из реального кода проекта; список действий —
шаблон, что искать. Подсистемы за пределами списка ищи тем же способом (см. «Новые подсистемы»).

### REST API
- Запрос на эндпоинт каждым методом, который используют тесты: GET/POST/PUT/PATCH/DELETE.
- Часто атом = «вызов конкретного API ресурса с авторизацией» (если есть типизированные клиенты
  ресурсов), а не сырой HTTP. Смотри, как тесты на самом деле зовут API.
- Отдельные атомы: получение/обновление токена авторизации; проверка статус-кода и десериализация
  тела в модель; работа с заголовками/query-параметрами, если это вынесено в обёртку.

### Kafka
- Отправить запись (с ключом/заголовками) в топик.
- Прочитать/дождаться записи из топика по условию (poll + фильтр + таймаут).
- Создать/очистить топик, сместить offset, если такое есть в инфраструктуре тестов.
- Сериализация/десериализация значения (Avro/JSON/Schema Registry), если вынесена в обёртку.

### PostgreSQL
- Выполнить SQL-запрос и получить строки (select).
- Выполнить update/insert/delete (DML) и получить число затронутых строк.
- Проверить наличие/значение строки (если есть готовый ассерт-хелпер).
- Подготовка/очистка данных (truncate, seed), если вызывается из тестов.
- Выполнение в транзакции/откат, если поддержано инфраструктурой.

### Ignite
- Получить значение из кэша по ключу.
- Положить/обновить значение в кэше.
- Удалить/инвалидировать запись; очистить кэш.
- Запрос по кэшу (SQL/ScanQuery), если используется.
- Дождаться появления/изменения значения (часто async).

### OpenSearch (поиск логов)
- Найти лог-записи по запросу (сервис/индекс/временное окно/текст).
- Дождаться появления строки лога по условию (поллинг с таймаутом — типичный async-атом).
- Подсчитать совпадения / проверить отсутствие записи.
- Построить запрос по корреляционному id/полю, если вынесено в обёртку.

### Kubernetes
- Получить логи пода/деплоймента (с фильтром по контейнеру/времени/строке).
- Рестартнуть под/деплоймент (delete pod / rollout restart) и дождаться готовности.
- Изменить настройку приложения (patch ConfigMap/Secret/env, scale) и применить.
- Дождаться состояния (под Ready, rollout завершён).
- Выполнить команду в поде (exec), проброс портов — если используется тестами.

## Новые подсистемы («и многое другое»)

Не ограничивайся списком выше. На Этапе 1 ты видел зависимости и слой клиентов — для **каждого**
найденного клиента/обёртки, к которому ходят тесты, заведи свои атомы по тем же правилам. Частые
кандидаты: gRPC, SOAP/legacy, Redis, S3/MinIO, RabbitMQ, scheduler/cron-триггеры, feature-flags,
сервис аутентификации/SSO, генераторы тестовых данных, работа с файлами/отчётами.

Эвристика поиска новой подсистемы:
1. Есть отдельный `*Client`/`*Service`/`*Steps`-класс под технологию.
2. Есть её зависимость в сборке и конфиг/контейнер в окружении.
3. Тесты реально к ней обращаются.
Если все три — это подсистема, заведи под неё раздел в каталоге.

## Что зафиксировать по каждому атому (для каталога)

Запиши в `_catalog.md` (формат — `templates/capability-catalog.template.md`):
- **подсистема**;
- **возможность** (короткое действие);
- **точка входа** — `Класс.метод(сигнатура)`;
- **`file:line`** точки входа;
- **нужный конфиг/env** (что должно быть настроено);
- **пример теста** — `file:line`, где это уже используется (источник сниппета на Этапе 4);
- **предлагаемое имя скилла** — `<подсистема>-<действие>` (см. соглашения в `SKILL.md`);
- **приоритет** — насколько часто встречается / насколько важно (для подтверждения на Этапе 3).

После заполнения каталога переходи к Этапу 3 (подтверждение с пользователем).
