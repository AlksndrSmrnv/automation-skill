# Autotest Skill Generator

Мета-скилл. Берёт существующий проект автотестов (Kotlin 2.2.20 / Java 21 / JUnit5 / Allure)
и превращает его функционал в **библиотеку скиллов**: по одному скиллу на каждую атомарную
возможность (отправить запись в Kafka, выполнить SQL, получить логи пода, найти строку в
OpenSearch, рестартнуть deployment, сделать REST-запрос с авторизацией и т.д.), плюс один
обзорный скилл стиля проекта.

## Платформа: Qwen Code CLI

Этот скилл и всё, что он генерирует, рассчитаны на **Qwen Code CLI** (не Claude Code). Учитывай
особенности Qwen:

- **Где живут скиллы:** `~/.qwen/skills/<имя>/SKILL.md` (персональные, для всех проектов) или
  `.qwen/skills/<имя>/SKILL.md` (внутри репозитория, шарятся через git). Поэтому **генерируемые
  скиллы по умолчанию пиши в `.qwen/skills/`** репозитория тестов — тогда Qwen увидит их сам.
- **Frontmatter:** обязательны только `name` и `description`. `name` — по маске
  `^[\p{L}\p{N}_:.-]+$` (буквы/цифры/`_`/`:`/`.`/`-`, без пробелов). Опционально `paths:` —
  glob-маски, которые включают скилл только при работе с подходящими файлами
  (например, `paths: ['**/src/test/**/*.kt']`). Поля `license`/`metadata` не нужны.
- **Срабатывание:** скиллы model-invoked — Qwen сам решает по `description`, когда подключить
  скилл; плюс пользователь может вызвать явно через `/skills <имя>`. Значит `description` должно
  быть конкретным и богатым на естественные формулировки шага теста, с RU/EN-триггерами и
  названиями технологий — это главный механизм триггера.
- **Сабагенты:** в Qwen нет Claude-специфичных `Explore`/`general-purpose`, но свои сабагенты
  поддерживаются (через `/agents` и Task Tool, конфиги в `.qwen/agents/`). Если подходящие
  сабагенты доступны — делегируй им разбор проекта по зонам; иначе анализируй сам инструментами
  чтения/поиска (read/grep/glob/shell).
- **Вопросы пользователю:** не используй Claude-style `AskUserQuestion`. В Qwen применяй
  `ask_user_question`, если он доступен; иначе спрашивай обычным текстом.

## Зачем это нужно (важно понимать цель)

Эта библиотека скиллов — вход для будущего агента-оркестровщика, который читает тест в виде
текстовых шагов и пишет реализацию **точно в стиле проекта**. Поэтому каждый сгенерированный
скилл должен быть таким, чтобы по нему можно было написать шаг теста, **не подглядывая в
исходники**: с точной сигнатурой, рабочим сниппетом и ссылкой на реальный код. Если скилл
заставляет читателя всё равно лезть в исходники — он недоделан.

Главный принцип: **ничего не выдумывай**. Всё содержимое скиллов выводится из реального кода
проекта. Каждый сниппет снят с живого теста, каждая сигнатура проверена, каждая ссылка
`file:line` существует. Лучше меньше скиллов, но достоверных, чем много правдоподобных выдумок.

## Предусловия

1. **Корень репозитория тестов.** Спроси у пользователя путь к проекту автотестов, если он не
   очевиден из текущей рабочей директории. Без доступа к коду генерация невозможна — этот скилл
   не работает «по памяти».
2. **Куда писать результат.** По умолчанию — `.qwen/skills/` в репозитории тестов (тогда Qwen
   обнаружит скиллы автоматически). Альтернатива — `~/.qwen/skills/` для доступа из всех проектов.
   Подтверди путь на Этапе 3.

## Рабочий процесс (6 этапов)

Не пропускай этапы и не перескакивай к генерации, пока не построил карту и не подтвердил каталог
с пользователем. Спешка здесь = скиллы, которые врут о коде.

### Этап 1. Картирование фреймворка

Цель — понять архитектуру проекта: модули, слои, базовые классы, конфиг, DI, стиль Allure и
ассертов. **Прочитай `references/discovery-guide.md`** и следуй методике.

Это широкий поиск по многим файлам. Если в Qwen доступны сабагенты (через `/agents` и Task Tool,
конфиги в `.qwen/agents/`) — делегируй им разбор по зонам: слои клиентов/степов и базовые классы →
конфиг/DI/Testcontainers → как устроены сами тест-классы и Allure-разметка. Если подходящих
сабагентов нет — разбирай эти зоны сам инструментами чтения/поиска (read/grep/glob/shell).
Собери выводы в единую карту проекта (держи её в рабочем контексте — на неё опираются Этапы 2–4).

Ключевой вывод этапа: **какой слой реально вызывают тесты** (обычно `steps`/хелперы с `@Step`,
а не сырые клиенты). Документировать на Этапе 4 нужно именно этот слой — иначе сгенерированные
тесты не совпадут со стилем проекта.

### Этап 2. Каталог атомарных возможностей

Цель — список атомарных возможностей, которые встречаются в тестах. **Прочитай
`references/capability-taxonomy.md`**, чтобы понять, что считать «атомом» в каждой подсистеме и
как распознавать подсистемы за пределами named-списка (REST/Kafka/Postgres/Ignite/OpenSearch/K8s).

Для каждой возможности зафиксируй: подсистему, точку входа (`class.method`), `file:line`, нужный
конфиг/env, пример реального теста (`file:line`), предлагаемое имя скилла, приоритет. Запиши
каталог в файл по формату `templates/capability-catalog.template.md`. Сохраняй его **вне**
skills-root, чтобы не мусорить тем, что Qwen сканирует как скиллы, — например
`.qwen/skill-catalog/_catalog.md`.

**Каталог тоже пишется в репозиторий — секреты в него не заноси.** В колонке «Конфиг/env»
указывай только ключи конфига и имена переменных (`kafka.bootstrap-servers`, `${DB_PASSWORD}`)
или плейсхолдеры, но не реальные значения, хосты/URL стендов, IP, namespace, токены, пароли.

Правила выделения атомов:
- Атом = одно осмысленное действие в тесте, переиспользуемое в нескольких тестах. Не сырой
  HTTP-вызов, а «GET ресурса X с авторизацией», если так делают тесты.
- Если действие нигде в тестах не вызывается напрямую — это не атом, а внутренняя деталь. Не
  документируй приватные/внутренние методы, до которых тесты не дотягиваются.
- Не дублируй: если две обёртки делают одно и то же, выбери ту, которую реально используют тесты.

### Этап 3. Подтверждение с пользователем

Полный каталог уже лежит в файле (Этап 2) — **не вываливай его целиком в чат**. В чат выведи
только краткую сводку: по каждой подсистеме число возможностей и 1–2 примера, плюс путь к файлу
каталога. Затем задай пользователю вопросы — через `ask_user_question`, если он доступен в Qwen,
иначе обычным текстом (Claude-style `AskUserQuestion` не используй):
- какие возможности генерировать (все / подмножество / по приоритету);
- что упущено или названо неверно;
- куда писать скиллы (подтверди путь вывода, по умолчанию `.qwen/skills/`).

Не начинай генерацию, пока пользователь не подтвердил каталог. Это дешёвая точка коррекции до
того, как ты создашь десятки файлов.

### Этап 4. Генерация скиллов

Сначала создай **обзорный скилл стиля** `autotest-house-style` по
`templates/house-style-skill.template.md` — он описывает базовый тест-класс, DI/конфиг, нейминг,
Allure-разметку, библиотеку ассертов, билдеры данных, жизненный цикл и то, как тест собирается
из возможностей end-to-end. Будущий оркестратор грузит его первым.

Затем по каждой подтверждённой возможности создай отдельный скилл по
`templates/capability-skill.template.md`. Каждый — в своей папке: `<output>/<имя-скилла>/SKILL.md`.

При заполнении сниппетов **копируй из реальных тестов** и минимально адаптируй (убери лишний
контекст, оставь рабочее ядро + Allure-шаг). Не сочиняй API, которого нет.

**Никогда не переноси секреты и чувствительные значения в скиллы.** Сгенерированные скиллы
обычно лежат в `.qwen/skills/` и шарятся через git — туда нельзя попадать токенам, паролям,
ключам, реальным хостам/URL стендов, IP, namespace/имён подов, e-mail и любым персональным или
внутренним данным. Везде (в сниппетах, таблицах параметров, разделе конфиг/env) заменяй такие
значения плейсхолдерами: `<TOKEN>`, `<DB_PASSWORD>`, `<KAFKA_BOOTSTRAP>`, `<BASE_URL>`,
`${ENV_VAR}` или ссылкой на ключ конфига (`kafka.bootstrap-servers`), но **не на само значение**.
Если сомневаешься, секрет ли это, — считай, что да, и заменяй.

### Этап 5. Самопроверка

**Прочитай `references/quality-checklist.md`** и пройди по нему каждый сгенерированный скилл:
все `file:line` существуют, сниппет ссылается на реальные символы и импорты, описание триггерится,
нет дублей, проставлены перекрёстные ссылки по имени, покрыты все подтверждённые возможности, обзорный
скилл создан. Исправь найденное до завершения.

### Этап 6. Итоговый отчёт

В конце выведи пользователю краткий отчёт:
- **путь к каталогу** (`_catalog.md`) и путь, куда записаны скиллы;
- **созданные скиллы** — список имён (включая `autotest-house-style`);
- **исключённые / черновые возможности** — что не сгенерировано и почему;
- **нерешённые TODO** — где не хватило данных или нужна проверка пользователем;
- **что проверено** — какие пункты `quality-checklist.md` пройдены (особенно: `file:line`
  резолвятся, секретов нет).

## Соглашения по именованию скиллов

Имя — kebab-case по схеме `<подсистема>-<действие>`:

| Подсистема | Примеры имён |
|------------|--------------|
| REST | `rest-get-with-auth`, `rest-post-json` |
| Kafka | `kafka-send-record`, `kafka-consume-await` |
| PostgreSQL | `postgres-run-query`, `postgres-assert-row` |
| Ignite | `ignite-cache-get`, `ignite-cache-put` |
| OpenSearch | `opensearch-find-log`, `opensearch-await-log` |
| Kubernetes | `k8s-get-pod-logs`, `k8s-restart-deployment`, `k8s-patch-config` |

Содержимое скиллов — **на русском**; код, имена символов и импорты остаются как в проекте
(английские).

## Структура этого скилла

```
autotest-skill-generator/

├── references/
│   ├── discovery-guide.md            ← методика картирования (Этап 1)
│   ├── capability-taxonomy.md        ← что такое «атом» по подсистемам (Этап 2)
│   └── quality-checklist.md          ← самопроверка (Этап 5)
└── templates/
    ├── capability-skill.template.md  ← шаблон скилла-возможности (Этап 4)
    ├── house-style-skill.template.md ← шаблон обзорного скилла (Этап 4)
    └── capability-catalog.template.md← формат каталога (Этап 2)
```

## Как запустить (в Qwen Code CLI)

1. Установи этот скилл: скопируй папку `autotest-skill-generator/` в `~/.qwen/skills/`
   (персонально) или в `.qwen/skills/` репозитория тестов.
2. Запусти Qwen в репозитории тестов и вызови скилл — явно `/skills autotest-skill-generator`
   или просто попроси:

```
Сгенерируй скиллы по нашему тест-фреймворку. Корень проекта автотестов: <путь>.
```

Ожидаемый результат: построена карта проекта → создан `_catalog.md` → после подтверждения
в `.qwen/skills/` появились папки скиллов (включая `autotest-house-style/`), Qwen видит их в
`/skills`, в каждом скилле сниппеты и ссылки указывают на реальные `file:line` проекта →
выдан итоговый отчёт (Этап 6).

## Чего не делать

- Не запускать генерацию без доступа к коду — скилл по памяти бесполезен и вреден.
- Не документировать слой, который тесты не вызывают (сырые клиенты вместо степов).
- Не выдумывать сигнатуры, импорты, конфиг — только то, что есть в репозитории.
- Не раздувать `SKILL.md` сгенерированных скиллов: детали и длинные примеры — в их собственные
  `references/`, если нужно (тот же принцип progressive disclosure).
